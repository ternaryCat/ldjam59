# ldjam59 — tower defence

Игра для Ludum Dare Jam #59 на Godot 4.6 (GL Compatibility, 2D). 
Тема джема: signals

## Идея

Делаем tower defence, но с обыми механиками:
	- игра делится на волны, каждая волная включает подготовку башен и отбитие от врагов
	- подготовка заключается в строительстве башен, это происходит неспешно
	- во время игры монстры будут появляться из определенного места, разбегаться по карте и атаковать мирные здания, игрок проиграет, если они будут разрушены
	- игрок будет играть персонажем, который будет передвигаться между противников и башен, он имеет возможность активировать поле вокруг себя, которое заставит противников следовать за ним под башни
	- игра планируется играться быстро, игрои и противники должны двигаться быстро
	- игрок за каждую волну будет получать деньги, но чем больше противники атакуют город, тем меньше денег достанется игроку
	- башни могут стрелять по противнику, но нет преследования снарядами противников, из-за этого башни могут промахиваться, что добавляет азарта и интереса

## Структура проекта

```
components/
  <feature>.tscn          # сцена-контейнер фичи верхнего уровня
  <feature>/              # всё, что принадлежит фиче: саб-сцены, скрипты, ассеты
	<node>.tscn
	<node>.gd
```

Правила разбиения:
- Один компонент = одна папка в `components/`. Снаружи торчит только `.tscn` фичи.
- Саб-узлы фичи живут внутри её папки (`components/level/player.gd` + `player.tscn`).
- Общие штуки (шейдеры, утилиты, автозагрузки) — добавляем папку рядом только когда реально появляется второй потребитель. До этого живёт внутри своей фичи.
- Ассеты (спрайты, звуки) — рядом со сценой, которая их использует. Перекладываем в общий `assets/` только при повторном использовании.

## Правила кода (GDScript)

- Скрипт и сцена для узла называются одинаково: `player.gd` ↔ `player.tscn`.
- `snake_case` для файлов, переменных, функций. `PascalCase` для классов и `class_name`. `SCREAMING_SNAKE_CASE` для констант.
- Типизируем аргументы и возвраты: `func take_damage(amount: int) -> void`. `@export` с типом.
- Коммуникация между узлами — через сигналы, а не через прямые ссылки вверх по дереву. Дети не знают про родителя.
- Никакого `get_node("../../Foo")`. Ссылки только через `@onready var x := $Child` или `@export`.
- Не пишем комментарии к очевидному (`# Called every frame`). Оставляем комментарий только если без него поведение неочевидно.
- Не добавляем систему/абстракцию ради одного кейса. Появился второй — тогда вытаскиваем.
- `_process` только для кадрового кода. Физика — в `_physics_process`. Ввод — в `_input`/`_unhandled_input`.

## Рабочий процесс

- Маленькие коммиты, зелёный `master` после каждого.
- Перед коммитом — открыть сцену в редакторе и убедиться, что не сломано (ошибки в `Output`, варнинги в `Debugger`).
- Когда добавляем новый компонент — сначала делаем `.tscn` + пустой `.gd`, потом наполняем.
- Если что-то временно — помечаем `# TODO(jam):` чтобы легко найти перед сдачей.

## Что делает Claude

- По умолчанию редактирует существующие файлы, не плодит новые.
- Перед крупной фичей — коротко сверяет план в чате.
- Не трогает `project.godot` и `.godot/` без явной просьбы (там сидят UID'ы и настройки редактора).
- При правках сцен (`.tscn`) старается менять минимум строк — формат чувствителен к порядку ресурсов.
